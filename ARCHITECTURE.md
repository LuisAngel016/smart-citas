## Capas y responsabilidades

### üéØ Domain (L√≥gica de Negocio Pura)
**No debe depender de frameworks ni HTTP**

- **entities/**: Modelos del dominio con su l√≥gica
  - Ejemplo: `appointment.entity.ts`, `client.entity.ts`
  - Representan conceptos del negocio
  - Pueden tener m√©todos de validaci√≥n o comportamiento

- **interfaces/**: Contratos y tipos compartidos
  - Ejemplo: `appointment-page.interface.ts`
  - Define estructuras de datos complejas
  - Tipos de respuesta paginadas

- **repositories/**: Interfaces de repositorios (contratos)
  - Ejemplo: `appointment.repository.ts`, `client.repository.ts`
  - Define √öNICAMENTE los contratos (interfaces)
  - NO contiene implementaciones

**Regla de oro:** Domain NO conoce c√≥mo se obtienen los datos, solo define QU√â datos necesita.

### üîÑ Application (Casos de Uso)

## Flujo de datos completo (ejemplo: crear una cita)

### üîÑ Paso a paso con archivos reales

1. **Usuario interact√∫a con UI** 
   ```
   üìÑ AppointmentsPage.tsx (presentation/pages/)

   ```

   ```
   ‚Üì
   ```


5. **Implementaci√≥n del repositorio hace llamada HTTP**
   ```
   üìÑ appointment.repository.impl.ts (infrastructure/repositories/)
   ‚Üì
   - Transforma entity ‚Üí DTO con mapper
   - Hace POST a la API
   - Recibe respuesta
   ```

6. **Mapper transforma respuesta API ‚Üí Entity**
   ```
   üìÑ appointment.mapper.ts (infrastructure/mappers/)
   ‚Üì
   - Recibe: AppointmentApiResponse (DTO)
   - Convierte: snake_case ‚Üí camelCase

   - Retorna: Appointment (Entity)
   ```

7. **Resultado sube por las capas**
   ```
   Mapper ‚Üí Repository ‚Üí UseCase ‚Üí Hook ‚Üí Component
   ‚Üì
   Actualiza UI con nueva cita creada
   ```
